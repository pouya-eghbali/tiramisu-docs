# MCP Server Redesign — HTTP JSON-RPC Endpoint

## Problem

The current MCP server is a local stdio-based tool that reads `.tiramisu` files from disk. This only benefits the docs author — not the users of the documentation site. A documentation site's MCP server should be hosted and accessible to anyone using AI assistants.

## Design

### Architecture

A single SvelteKit POST route at `/mcp` that speaks JSON-RPC 2.0. No MCP SDK dependency at runtime — just a request handler that parses JSON-RPC, dispatches to tool functions, and returns results. Works on every deployment target including serverless.

### Config

```typescript
interface TiramisuDocsConfig {
  mcp?: boolean | string
  // true    → scaffolder generates /mcp route, UI buttons use config.url + "/mcp"
  // string  → no route generated, UI buttons point to the custom URL
  // omitted → no MCP, no buttons
}
```

### Route: `src/routes/mcp/+server.ts`

Generated by scaffolder. Handles POST with JSON-RPC 2.0:

- `initialize` → server info + capabilities (`{ tools: {} }`)
- `tools/list` → 5 tool definitions with JSON Schema inputs
- `tools/call` → dispatches to tool handler, returns result

Data from `virtual:tiramisu-docs` — no disk scanning.

### Kit: `handleMcpRequest()` helper

Pure function exported from `@tiramisu-docs/kit/mcp`:

```typescript
import { handleMcpRequest } from "@tiramisu-docs/kit/mcp"
import { docs, searchIndex, sidebar } from "virtual:tiramisu-docs"

export async function POST({ request }) {
  const body = await request.json()
  const result = handleMcpRequest(body, { docs, searchIndex, sidebar })
  return new Response(JSON.stringify(result), {
    headers: { "Content-Type": "application/json" }
  })
}
```

### Tools (5, unchanged from current)

| Tool | Input | Source |
|------|-------|--------|
| `search_docs` | `{ query, limit? }` | searchIndex |
| `read_doc` | `{ slug }` | searchIndex + docs |
| `list_pages` | `{ section? }` | docs |
| `list_sections` | `{}` | sidebar |
| `get_table_of_contents` | `{ slug }` | docs |

### Open Links

`getOpenLinks()` accepts `mcp?: boolean | string`:
- `true` → derives URL from `baseUrl + "/mcp"`
- string → uses the string directly
- "Connect with MCP" copies the URL
- "Connect to VSCode" uses `vscode:mcp/install?` with the URL

### Removals

- `packages/kit/src/mcp.ts` (stdio server)
- `packages/kit/src/bin/mcp.ts` (CLI)
- `packages/kit/src/scan.ts` (disk scanning, if unused elsewhere)
- `bin` field from kit `package.json`
- `mcp` script from scaffolder package.json template
- `@modelcontextprotocol/sdk` dependency

### Additions/Modifications

- `packages/kit/src/mcp.ts` — rewritten as `handleMcpRequest()` pure function
- `packages/kit/src/config.ts` — `mcp` type → `boolean | string`
- `packages/kit/src/lib/open-links.ts` — handle `boolean | string`
- Scaffolder: `/mcp` route template, updated prompts, removed `mcp` script
- Playground: `/mcp` route
- Docs: rewritten MCP page
